# meshdb

This is an R package that is used to traverse the [MeSH medical ontology](https://pubmed.ncbi.nlm.nih.gov/help/#using-mesh-database)


## Project State

`meshdb` is an R package for navigating a local MeSH ontology database stored as parquet files. The package opens an in-memory DuckDB database, registers parquet files as temporary views, and uses dbplyr/dplyr pipelines so queries stay lazy until collected.

The public API currently covers:

- `meshdb()` / `mesh_disconnect()`: create and close a DuckDB-backed object.
- `mesh_data_path()` / `mesh_tables()`: resolve the parquet directory and required file paths.
- `mesh_lookup()`: find MeSH terms by ID, term, synonym, or qualifier.
- `mesh_parents()`, `mesh_children()`, `mesh_ancestors()`: traverse direct and transitive MeSH relationships.
- `mesh_common_parents()`: find shared direct parents or ancestors for a set of terms.
- `mesh_is_child()`: test whether MeSH IDs descend from a single parent term and report distance.
- `mesh_map()`: query human or mouse PubMed-to-MeSH mapping tables with optional PubMed, MeSH, and gene filters.

The default data directory is controlled by `MESHDB_PARQUET_PATH` environment variable; if this is not defined, the package warns the user when the package is loaded.

Required parquet files:

- `meshdb.parquet` -> registered as table `meshdb`
- `parent.parquet` -> registered as table `parent`
- `ancestor.parquet` -> registered as table `ancestor`
- `meshmap-human.parquet` -> registered as table `human`
- `meshmap-mouse.parquet` -> registered as table `mouse`

Expected core columns:

- Terms: `mesh_id`, `mesh_term`, `category_code`, `category`, optionally `synonym`, `qualifier_id`, `qualifier_term`
- Direct relationships: `parent_id`, `child_id`, `category_code`, `category`
- Ancestors: `ancestor_id`, `offspring_id`, `category_code`, `category`
- Mapping tables: `species`, `pmid`, `mesh_id`, `mesh_term`, `category_code`, `category`, `ensembl_id`, `entrez_id`, `gene`, `gene_biotype`, `sourcedb`

## Implementation Notes

- Prefer keeping operations lazy with dbplyr until the public function returns a tibble.
- Use `mesh_tbl(db, table)` to validate a `meshdb` object and access registered views.
- Use `resolve_mesh_ids()` when accepting either MeSH IDs or exact term labels.
- Relationship helpers annotate IDs through `term_summary()` and `annotate_terms()`.
- `mesh_is_child()` computes parent distance from direct `parent` edges with a recursive DuckDB query, so it does not require a distance column in `ancestor.parquet`.
- `copy_resolved_ids()` copies resolved inputs into a temporary DuckDB table so joins can happen in the database.
- `NAMESPACE` and `man/*.Rd` are generated by roxygen2; edit roxygen comments in `R/*.R` instead of editing generated files directly.

## Tests

Tests use `tests/testthat/helper-data.R` to generate small temporary parquet fixtures with `arrow::write_parquet()`. This keeps tests independent of the full local MeSH data directory.

Current test coverage includes:

- DuckDB-backed object creation and table registration.
- Lookup by ID, term label, and synonym.
- Parent, ancestor, and shared-parent relationship traversal.
- Child/ancestor membership and distance checks.
- PubMed mapping filters by species, MeSH term, and gene.

Run the package tests with:

```sh
Rscript -e 'testthat::test_local()'
```

If dependencies are missing, install the R package dependencies listed in `DESCRIPTION` before rerunning tests. The test suite requires suggested packages `arrow`, `testthat`, and `withr`.

## Style And Maintenance

- Keep package code in `R/` and tests in `tests/testthat/`.
- Follow the existing tidyverse style: base pipe `|>`, explicit `.data$` pronouns inside dplyr verbs, and `rlang::abort()`/`rlang::warn()` for user-facing conditions.
- Preserve the lightweight package shape. Add abstractions only when they reduce repeated query logic or clarify a shared data contract.
- Avoid requiring the full production parquet directory in tests; extend the temporary fixture data instead.
- This directory may not be a Git repository in the current workspace, so use filesystem inspection rather than relying on Git history.
